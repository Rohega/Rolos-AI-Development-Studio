# US-011: Confirm a receipt

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Priority:** P0  
**Dependencies:** US-010, US-005

---

## Goal

Record the goods physically received, assign a location, and update the warehouse inventory with full traceability.

## User

**Operator** (`operario`), **Supervisor** (`supervisor`)

## Description

As an operator, I want to confirm the quantities received per line of the receiving order and assign a destination location, so I can increase the available stock in the warehouse.

## Acceptance criteria

- [ ] I can confirm the received quantity per line (≤ or > the expected quantity)
- [ ] I must indicate a destination location for each confirmed line
- [ ] When confirmed, `quantity_on_hand` increases via `StockUpdater` and a `stock_movement` of type `reception` is created
- [ ] A partial receipt leaves the order in `partial`; completing all lines → `completed`
- [ ] A discrepancy (received ≠ expected) is flagged visually without blocking
- [ ] I cannot confirm in an inactive warehouse

## Business rules

1. Confirmation does not require supervisor approval (it is the normal operational flow).
2. The location must belong to the order's warehouse and be active.
3. Received quantity ≥ 0; 0 means a line not received in this batch.
4. Stock is never negative (only increments during receiving).
5. Auditable movement with user and timestamp.

## Edge cases

| Case | Expected behavior |
|------|-------------------------|
| Received > expected | Allow it; flag positive discrepancy |
| Received < expected | Allow it; the order stays `partial` |
| Confirm the same line twice | Idempotency: a second confirmation adds or blocks depending on the design → **MVP: one confirmation per line; edit before closing the order** |
| Location with a different product | Allow multiple products per location |
| Two operators on the same line concurrently | Optimistic lock on the line; the second one gets an error |

## Preliminary technical notes

- `Reception::ConfirmLine.call(line, received_qty, location, user)` service
- Transaction: update the line + `StockUpdater` + movement
- Tablet-friendly UI: product search by SKU/name
- Cumulative `received_quantity` field on the line
- Branch: `feature/warehouse-reception`

## Versión en español

# US-011: Confirmar recepción

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Prioridad:** P0  
**Dependencias:** US-010, US-005

---

## Objetivo

Registrar la mercancía físicamente recibida, asignar ubicación y actualizar el inventario del almacén con trazabilidad completa.

## Usuario

**Operario** (`operario`), **Supervisor** (`supervisor`)

## Descripción

Como operario, quiero confirmar las cantidades recibidas por línea de la orden de recepción y asignar ubicación de destino, para incrementar el stock disponible en el almacén.

## Criterios de aceptación

- [ ] Puedo confirmar cantidad recibida por línea (≤ o > cantidad esperada)
- [ ] Debo indicar ubicación de destino por línea confirmada
- [ ] Al confirmar, `quantity_on_hand` aumenta vía `StockUpdater` y se crea `stock_movement` tipo `reception`
- [ ] Recepción parcial deja orden en `partial`; completar todas las líneas → `completed`
- [ ] Discrepancia (recibido ≠ esperado) se marca visualmente sin bloquear
- [ ] No puedo confirmar en almacén inactivo

## Reglas de negocio

1. Confirmación no requiere aprobación de supervisor (salida normal de flujo operativo).
2. Ubicación debe pertenecer al almacén de la orden y estar activa.
3. Cantidad recibida ≥ 0; 0 equivale a línea no recibida en esta tanda.
4. Stock nunca negativo (solo incrementos en recepción).
5. Movimiento auditable con usuario y timestamp.

## Casos borde

| Caso | Comportamiento esperado |
|------|-------------------------|
| Recibido > esperado | Permitir; flag discrepancia positiva |
| Recibido < esperado | Permitir; orden queda `partial` |
| Confirmar misma línea dos veces | Idempotencia: segunda confirmación suma o bloquea según diseño → **MVP: una confirmación por línea; editar antes de cerrar orden** |
| Ubicación con producto distinto | Permitir múltiples productos por ubicación |
| Concurrencia dos operarios misma línea | Lock optimista en línea; segundo recibe error |

## Notas técnicas preliminares

- Servicio `Reception::ConfirmLine.call(line, received_qty, location, user)`
- Transacción: actualizar línea + `StockUpdater` + movimiento
- UI tablet-friendly: búsqueda producto por SKU/nombre
- Campo `received_quantity` acumulativo en línea
- Rama: `feature/warehouse-reception`
