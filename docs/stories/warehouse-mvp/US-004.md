# US-004: Location management

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Priority:** P0  
**Dependencies:** US-003

---

## Goal

Define physical locations inside each warehouse to know where each product is stored and to optimize picking routes.

## User

**Administrator** (`admin`)

## Description

As an administrator, I want to create locations inside a warehouse (aisle, rack, position), so I can assign stock to specific spots and generate ordered picking lists.

## Acceptance criteria

- [ ] I can create locations linked to a warehouse
- [ ] Each location has a readable code that is unique within the warehouse (e.g. `A-01-03`)
- [ ] Fields: aisle (`aisle`), rack (`rack`), position (`position`), and a composite code
- [ ] I can deactivate a location with no stock
- [ ] I cannot deactivate or delete a location with `quantity_on_hand > 0`
- [ ] List filterable by warehouse and status

## Business rules

1. Code unique per `(warehouse_id, code)`.
2. A location belongs to exactly one warehouse.
3. An inactive location does not accept new stock assignments.
4. The code can be generated automatically from aisle/rack/position or entered manually (implementation decision: auto from components).

## Edge cases

| Case | Expected behavior |
|------|-------------------------|
| Stock in a location when deactivating | Block deactivation |
| Two locations with the same code in different warehouses | Allowed |
| Location with no stock but with historical movements | Allow deactivation |
| Change the code while stock is present | Rejected in the MVP |

## Preliminary technical notes

- `Location` model: `warehouse_id`, `code`, `aisle`, `rack`, `position`, `active`
- Composite unique index `(warehouse_id, code)`
- Callback or service to generate `code` = `"#{aisle}-#{rack}-#{position}"`
- API: `GET/POST/PATCH /api/warehouses/:id/locations`
- Picking will sort by `aisle`, `rack`, `position` lexicographically

## Versión en español

# US-004: Gestión de ubicaciones

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Prioridad:** P0  
**Dependencias:** US-003

---

## Objetivo

Definir ubicaciones físicas dentro de cada almacén para saber dónde se almacena cada producto y optimizar recorridos de picking.

## Usuario

**Administrador** (`admin`)

## Descripción

Como administrador, quiero crear ubicaciones dentro de un almacén (pasillo, estante, posición), para asignar stock a posiciones concretas y generar listas de picking ordenadas.

## Criterios de aceptación

- [ ] Puedo crear ubicaciones asociadas a un almacén
- [ ] Cada ubicación tiene código legible único dentro del almacén (ej. `A-01-03`)
- [ ] Campos: pasillo (`aisle`), estante (`rack`), posición (`position`) y código compuesto
- [ ] Puedo desactivar ubicación sin stock
- [ ] No puedo desactivar ni eliminar ubicación con `quantity_on_hand > 0`
- [ ] Listado filtrable por almacén y estado

## Reglas de negocio

1. Código único por `(warehouse_id, code)`.
2. Ubicación pertenece a exactamente un almacén.
3. Ubicación inactiva no acepta nuevas asignaciones de stock.
4. El código puede generarse automáticamente desde aisle/rack/position o ingresarse manualmente (decisión implementación: auto desde componentes).

## Casos borde

| Caso | Comportamiento esperado |
|------|-------------------------|
| Stock en ubicación al desactivar | Bloquear desactivación |
| Dos ubicaciones mismo código en almacenes distintos | Permitido |
| Ubicación sin stock, con movimientos históricos | Permitir desactivar |
| Cambiar código con stock presente | Rechazado en MVP |

## Notas técnicas preliminares

- Modelo `Location`: `warehouse_id`, `code`, `aisle`, `rack`, `position`, `active`
- Índice único compuesto `(warehouse_id, code)`
- Callback o servicio para generar `code` = `"#{aisle}-#{rack}-#{position}"`
- API: `GET/POST/PATCH /api/warehouses/:id/locations`
- Picking ordenará por `aisle`, `rack`, `position` lexicográficamente
