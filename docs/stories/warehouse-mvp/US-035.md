# US-035: Cancel a confirmed movement

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Priority:** P0  
**Dependencies:** US-011, US-022

---

## Goal

Allow already-confirmed operations (receiving, shipping, transfer) to be reversed only with supervisor approval, keeping inventory integrity and audit.

## User

**Operator** (requests), **Supervisor** / **Administrator** (approves)

## Description

As an operator, I want to request the cancellation of an already-confirmed stock movement, stating the reason, so a supervisor approves and the system generates the corresponding offsetting movement.

## Acceptance criteria

- [ ] I can request cancellation of a `stock_movement` that has not been cancelled before
- [ ] Reason required when requesting
- [ ] The request stays in `pending`; stock does **not** change until approval
- [ ] The supervisor can approve: creates an inverse offsetting movement, marks the original with `cancelled_at`, links `reversal_movement_id`
- [ ] The supervisor can reject with a comment
- [ ] The operator cannot approve their own request
- [ ] List of pending requests for supervisors

## Business rules

1. Only movements of these types: `reception`, `outbound`, `transfer_out`, `transfer_in`, `adjustment` (approved adjustments).
2. Do not cancel a movement that is already a reversal of another.
3. The reversal must keep stock ≥ 0 and stay consistent with reservations.
4. Cancelling a receipt decreases stock; cancelling a shipment increases it.
5. Cancelling a `transfer_out` in transit with no destination receipt: restores the source.
6. If the transfer was already received at the destination, cancellation requires an inverse flow (a transfer back) — **MVP: block simple cancellation; requires a manual inverse transfer**.

## Edge cases

| Case | Expected behavior |
|------|-------------------------|
| Cancel a shipment but the stock was already physically sold | The supervisor decides; operational risk — warning in the UI |
| Cancel a receipt without enough stock in the location | Reject the approval |
| Double request on the same movement | Reject the duplicate |
| Movement from >90 days ago | Allow with a warning |

## Preliminary technical notes

- `MovementCancellation` model: `stock_movement_id`, `status`, `requested_by`, `approved_by`, `reason`, `reversal_movement_id`
- `Cancellations::Request`, `Cancellations::Approve` service (creates an inverse movement via StockUpdater)
- Append-only: do not delete `stock_movements`
- Branch: `feature/warehouse-cancellations`

## Versión en español

# US-035: Cancelar movimiento confirmado

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Prioridad:** P0  
**Dependencias:** US-011, US-022

---

## Objetivo

Permitir revertir operaciones ya confirmadas (recepción, salida, transferencia) solo con aprobación de supervisor, manteniendo integridad del inventario y auditoría.

## Usuario

**Operario** (solicita), **Supervisor** / **Administrador** (aprueba)

## Descripción

Como operario, quiero solicitar la cancelación de un movimiento de stock ya confirmado, indicando el motivo, para que un supervisor apruebe y el sistema genere el movimiento compensatorio correspondiente.

## Criterios de aceptación

- [ ] Puedo solicitar cancelación sobre `stock_movement` no cancelado previamente
- [ ] Motivo obligatorio al solicitar
- [ ] Solicitud queda en `pending`; stock **no** cambia hasta aprobación
- [ ] Supervisor puede aprobar: crea movimiento compensatorio inverso, marca original con `cancelled_at`, enlaza `reversal_movement_id`
- [ ] Supervisor puede rechazar con comentario
- [ ] Operario no puede aprobar su propia solicitud
- [ ] Listado de solicitudes pendientes para supervisores

## Reglas de negocio

1. Solo movimientos de tipos: `reception`, `outbound`, `transfer_out`, `transfer_in`, `adjustment` (ajustes aprobados).
2. No cancelar movimiento que ya es reversión de otro.
3. Reversión debe mantener stock ≥ 0 y coherencia con reservas.
4. Cancelar recepción decrementa stock; cancelar salida incrementa.
5. Cancelar `transfer_out` en tránsito sin recepción destino: restaura origen.
6. Si transferencia ya recibida en destino, cancelación requiere flujo inverso (transferencia de vuelta) — **MVP: bloquear cancelación simple; requiere transferencia manual inversa**.

## Casos borde

| Caso | Comportamiento esperado |
|------|-------------------------|
| Cancelar salida pero stock ya vendido físicamente | Supervisor decide; riesgo operativo — advertencia en UI |
| Cancelar recepción sin stock suficiente en ubicación | Rechazar aprobación |
| Doble solicitud mismo movimiento | Rechazar duplicado |
| Movimiento de hace >90 días | Permitir con advertencia |

## Notas técnicas preliminares

- Modelo `MovementCancellation`: `stock_movement_id`, `status`, `requested_by`, `approved_by`, `reason`, `reversal_movement_id`
- Servicio `Cancellations::Request`, `Cancellations::Approve` (crea movimiento inverso vía StockUpdater)
- Append-only: no borrar `stock_movements`
- Rama: `feature/warehouse-cancellations`
