# US-003: Warehouse management

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Priority:** P0  
**Dependencies:** none

---

## Goal

Register the company's distribution centers/warehouses so inventory and operations can be split by site.

## User

**Administrator** (`admin`)

## Description

As an administrator, I want to create and manage warehouses with a code and name, so I can run stock and movements independently at each distribution center.

## Acceptance criteria

- [ ] I can create a warehouse with a unique code and a name
- [ ] I can record an optional address
- [ ] I can deactivate a warehouse
- [ ] An inactive warehouse does not allow new receipts, shipments, or transfers
- [ ] An inactive warehouse still allows stock and history lookups
- [ ] The list shows code, name, status, and total product stock (summary)

## Business rules

1. Warehouse code must be unique (e.g. `CD-MAD`, `CD-BCN`).
2. At least 1 active warehouse must exist in the system at all times.
3. Deactivating does not delete historical data or recorded stock.
4. Operators can be assigned to a default warehouse (P1 configuration; in the MVP they filter manually).

## Edge cases

| Case | Expected behavior |
|------|-------------------------|
| Deactivate the only active warehouse | Rejected |
| Deactivate a warehouse with transfers in transit | Rejected until they are completed or cancelled |
| Code in lowercase | Normalize to uppercase when saving |
| Reactivate a warehouse | Allowed; new operations enabled |

## Preliminary technical notes

- `Warehouse` model: `code`, `name`, `address`, `active`
- Unique index on `code`
- `ExternalReference` prepared for ERP mapping (e.g. `stock.warehouse`)
- API: `GET/POST/PATCH /api/warehouses`
- Policy: only `admin` can create/edit warehouses

## Versión en español

# US-003: Gestión de almacenes

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Prioridad:** P0  
**Dependencias:** ninguna

---

## Objetivo

Registrar los centros de distribución/almacenes de la comercializadora para segmentar inventario y operaciones por sede.

## Usuario

**Administrador** (`admin`)

## Descripción

Como administrador, quiero crear y gestionar almacenes con código y nombre, para operar stock y movimientos de forma independiente en cada centro de distribución.

## Criterios de aceptación

- [ ] Puedo crear un almacén con código único y nombre
- [ ] Puedo registrar dirección opcional
- [ ] Puedo desactivar un almacén
- [ ] Almacén inactivo no permite nuevas recepciones, salidas ni transferencias
- [ ] Almacén inactivo sí permite consulta de stock e historial
- [ ] El listado muestra código, nombre, estado y stock total de productos (resumen)

## Reglas de negocio

1. Código de almacén único (ej. `CD-MAD`, `CD-BCN`).
2. Mínimo 1 almacén activo en el sistema en todo momento.
3. Desactivar no elimina datos históricos ni stock registrado.
4. Operarios pueden estar asignados a un almacén por defecto (configuración P1; en MVP filtran manualmente).

## Casos borde

| Caso | Comportamiento esperado |
|------|-------------------------|
| Desactivar único almacén activo | Rechazado |
| Desactivar almacén con transferencias en tránsito | Rechazado hasta completar o cancelar |
| Código con minúsculas | Normalizar a mayúsculas al guardar |
| Reactivar almacén | Permitido; operaciones nuevas habilitadas |

## Notas técnicas preliminares

- Modelo `Warehouse`: `code`, `name`, `address`, `active`
- Índice único en `code`
- `ExternalReference` preparado para mapeo ERP (p. ej. `stock.warehouse`)
- API: `GET/POST/PATCH /api/warehouses`
- Política: solo `admin` puede crear/editar almacenes
