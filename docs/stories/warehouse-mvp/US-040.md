# US-040: User and role management

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Priority:** P0  
**Dependencies:** none

---

## Goal

Control access to the WMS based on the distributor's operational responsibilities.

## User

**Administrator** (`admin`)

## Description

As an administrator, I want to create users and assign them one of the defined roles (admin, supervisor, operator, viewer), so each person only accesses the functions that belong to them.

## Acceptance criteria

- [ ] I can create a user with email, name, and role
- [ ] I can deactivate a user; they cannot log in
- [ ] Available roles: `admin`, `supervisor`, `operario`, `consulta`
- [ ] Permission matrix implemented per the spec (RBAC)
- [ ] I cannot delete the last active `admin`
- [ ] Role change effective in the next session or immediately (MVP: immediate)

## Business rules

| Action | admin | supervisor | operario | consulta |
|--------|-------|------------|----------|----------|
| Catalog/warehouses CRUD | ✓ | — | — | — |
| Receiving/shipping/transfer | ✓ | ✓ | ✓ | — |
| Record count | ✓ | ✓ | ✓ | — |
| Approve adjustments/cancellations | ✓ | ✓ | — | — |
| Inventory lookup | ✓ | ✓ | ✓ | ✓ |
| User management | ✓ | — | — | — |

1. Password per the project policy (min. 8 characters).
2. A deactivated user keeps their audit history.

## Edge cases

| Case | Expected behavior |
|------|-------------------------|
| Self-downgrade from admin | Reject if it is the only admin |
| Duplicate email | Reject |
| Operator accesses an approval URL | 403 |
| Deactivated user with an active session | Invalidate the session on the next request |

## Preliminary technical notes

- `User` model with a `role` enum
- Pundit policies per resource/action
- Devise or equivalent for auth
- Seed: initial admin user
- Branch: `feature/warehouse-auth-rbac`

# US-040: Gestión de usuarios y roles

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Prioridad:** P0  
**Dependencias:** ninguna

---

## Objetivo

Controlar acceso al WMS según responsabilidades operativas de la distribuidora.

## Usuario

**Administrador** (`admin`)

## Descripción

Como administrador, quiero crear usuarios y asignarles uno de los roles definidos (admin, supervisor, operario, consulta), para que cada persona acceda solo a las funciones que le corresponden.

## Criterios de aceptación

- [ ] Puedo crear usuario con email, nombre y rol
- [ ] Puedo desactivar usuario; no puede iniciar sesión
- [ ] Roles disponibles: `admin`, `supervisor`, `operario`, `consulta`
- [ ] Matriz de permisos implementada según spec (RBAC)
- [ ] No puedo eliminar el último `admin` activo
- [ ] Cambio de rol efectivo en próxima sesión o inmediato (MVP: inmediato)

## Reglas de negocio

| Acción | admin | supervisor | operario | consulta |
|--------|-------|------------|----------|----------|
| Catálogo/almacenes CRUD | ✓ | — | — | — |
| Recepción/salida/transferencia | ✓ | ✓ | ✓ | — |
| Conteo registrar | ✓ | ✓ | ✓ | — |
| Aprobar ajustes/cancelaciones | ✓ | ✓ | — | — |
| Consulta inventario | ✓ | ✓ | ✓ | ✓ |
| Gestión usuarios | ✓ | — | — | — |

1. Contraseña según política del proyecto (mín. 8 caracteres).
2. Usuario desactivado mantiene historial de auditoría.

## Casos borde

| Caso | Comportamiento esperado |
|------|-------------------------|
| Auto-degradarse de admin | Rechazar si es único admin |
| Email duplicado | Rechazar |
| Operario accede URL de aprobación | 403 |
| Usuario desactivado con sesión activa | Invalidar sesión en próximo request |

## Notas técnicas preliminares

- Modelo `User` con enum `role`
- Pundit policies por recurso/acción
- Devise o equivalente para auth
- Semilla: usuario admin inicial
- Rama: `feature/warehouse-auth-rbac`
